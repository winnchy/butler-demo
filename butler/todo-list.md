### 必看：项目的实现逻辑

#### 项目结构
- ① butler/ — 管家的"大脑" ：
  - 告诉 AI：你是谁、用户是谁、遇到情况怎么做
- ② mock_backend/ — 管家的"眼睛和耳朵"：
  - 提供数据：天气、路况、餐厅、排队、航班……
  - 而且这些数据会自己变化（模拟真实世界）
- ③ OpenClaw（AI 智能体框架，一个"壳"）：
  - 把 ① 和 ② 连接起来，让管家真正"运行"起来

#### 核心概念
- FastAPI：
  - 一个用 Python 写 Web 服务器的工具。说人话：你在浏览器输入网址能打开网页，就是因为有 Web 服务器在背后工作。
  - 我们这个项目里，FastAPI 的作用是——当管家的 AI 需要"查一下今天的天气"，它就向 FastAPI 服务器发送一个请求（就像你在百度搜索框输入关键词），服务器返回天气数据。
- API 路由：
  - "网址 → 对应功能"的映射表。比如：
    - 访问 /api/weather/current → 返回当前天气
    - 访问 /api/dining/recommend → 返回餐厅推荐
- WorldState
  - 世界状态引擎。它是整个 mock_backend 的心脏。
  - WorldState = 世界状态引擎。它是整个 mock_backend 的心脏。
  - 真实世界的数据是不断变化的——刚才还有位的餐厅现在排了 15 桌，外面突然下暴雨了，航班延误了 2 小时。WorldState 就是模拟这种"变化"的。
  - 它每秒都在后台运行，每隔几分钟就自动更新：排队人数随机增减、天气按季节模式变化、路况跟随早晚高峰波动。这样当管家查询数据时，每次得到的结果都不一样——就像真实世界。
- Docker
  - 一个能把软件和其运行环境打包在一起的集装箱，有了它，你的程序就可以在任何地方保持一致
    - 想象一下你写了一个程序，在你的电脑上运行得完美无缺，但是一到别人得电脑或者服务器就报错：通常是因为环境不一致导致的，比如py版本、依赖库版本不同等等
    - Docker把程序及其所需要的所有依赖、配置文件、环境变量等都全部打包进一个叫“容器”的独立单元里。这样无论这个容器被放到开发、测试还是生产的服务器上，都能保证以相同的方式运行
  - 相关概念
    - image镜像：软件安装包/只读模板，包含了运行所需要的一切，比如代码、运行时库、环境变量和配置文件
    - container容器：镜像的运行实例，就像双击安装包以后运行起来的软件程序。你可以从一个镜像创建出多个相互隔离的容器
    - repository仓库：用来存放和分享镜像的地方，比如Docker Hub，上面有海量官方+用户自创的镜像

SOUL.md：管家的"人设文档"。定义了管家的性格（温暖但不肉麻、细心但不啰嗦）、行事风格（先人一步、面面俱到）、以及它如何服务三个不同用户。

USER.md / MEMORY.md：OpenClaw 标准文件。管家每次对话前会读取这两个文件来"认识"当前用户。切换用户就是把 profiles/ 下的不同模板复制到这里。

HEARTBEAT.md：管家的"心跳调度器"。定义了什么时候该主动做什么——比如每天早上 7:00 推送天气和通勤、周五 18:00 推送周末活动、每 2 分钟检查排队变化。它不直接跟用户说话，而是"唤醒"对应的 Skill 去执行。

SKILL.md：每个技能的"操作手册"。OpenClaw 的标准格式——文件头有 name 和 description，正文定义了这个技能在什么情况下触发、一步一步怎么执行、调用哪些 API、返回什么样的卡片。

profiles/：用户模板库。OpenClaw 本身只支持 1 对 1（一个管家对一个用户），但我们比赛要演示 3 个用户。解决方案：把 3 个用户的资料存在 profiles 里，演示时通过管理面板一键切换。

├── main.py              ← 服务器入口：启动一切
├── config.py            ← 全局配置：所有"知识"和规则
├── enrich_pois.py        ← 数据丰富化：给原始数据补全字段

├── api_routes/           ← API 接口：管家查询数据的"窗口"
│   ├── dining_routes.py  ← 餐饮相关接口（推荐/排队/预订…）
│   ├── mobility_routes.py← 出行相关接口（路线/票务/叫车…）
│   ├── city_routes.py    ← 活动相关接口（展览/促销/亲子…）
│   ├── outfit_routes.py  ← 天气+穿搭接口
│   ├── life_routes.py    ← 日程+记忆接口
│   └── admin_routes.py   ← 管理接口（切换用户/触发场景/面板）
├── data/
│   ├── beijing_seeds.json ← 原始种子数据（856 条真实北京 POI）
│   └── enriched/          ← 丰富化后的数据（启动时加载到内存）
└── fetch_all_seeds.py     ← 数据采集脚本（从高德 API 抓取）

main.py：服务器的主文件。python main.py 就是执行它。启动时会：

加载所有丰富化数据到内存
构建地铁网络和路网
启动 WorldState 后台引擎
挂载所有 API 路由
config.py：全局"知识库"。包含：60+ 种菜系关键词（用来从餐厅名字推断菜系）、北京 20 条地铁线路的完整站点、每个时段的拥堵指数、季节天气参数、21 个场景的触发器预设。所有其他文件都从它这里读取规则。

enrich_pois.py：数据丰富化管道。高德 API 只返回餐厅的基本信息（名字、坐标、地址）。但管家需要知道菜系、评分、人均消费、有没有包厢、能不能带宠物、最近的评价……这个脚本就是给原始数据"补全"这些字段。它不是随机乱填——比如菜系是从餐厅名字推断的（"阳坊大都涮肉" → 火锅），交通距离是用坐标算出来的（到最近地铁站真实多少米）。

route_generator.py：交通网络构建器。从 40 个真实地铁站坐标出发，匹配到北京 20 条线路，通过插值生成 371 个站点的完整网络。再加上模拟路网（环路+主干道+高速），实现多模式路径规划（地铁/驾车/骑行/步行/打车）。

world_state.py：世界状态引擎。用一个后台线程每 60 秒 tick 一次：

每 2 分钟：更新餐厅排队（随机游走 + 饭点偏向 + 天气影响）
每 15 分钟：更新路况（按时段拥堵曲线 + 随机事故）
每 30 分钟：更新商场/公园拥挤度（受天气联动影响）、检查优惠到期
每小时：更新天气（温度平滑移动 + 夏季午后雷暴 + 春季沙尘暴）、检查随机事件（乐乐发烧、布丁呕吐、航班延误、餐厅歇业、外卖延迟、地铁故障）
api_routes/：API 接口层。每个文件定义了一组"管家可以查询什么"：

dining_routes.py：12 个接口——推荐餐厅、查排队、取号、预订、紧急方案、外卖、评价……
mobility_routes.py：5 个接口——路径规划、票务查询、叫车、周边设施、长途规划
city_routes.py：4 个接口——活动推荐、商场促销、跳蚤市场、亲子活动
outfit_routes.py：7 个接口——当前天气、预报、预警、穿搭建议、突发预警、换季趋势、衣橱
life_routes.py：8 个接口——创建/查询日程、保存/搜索记忆、特殊日期、健康提醒
admin_routes.py：管理接口——切换用户、触发场景、查看状态、管理面板 HTML 页面
